# Contributing to Empire of Bits

Thank you for your interest in contributing to Empire of Bits! We're excited to have you join our community of developers, gamers, and blockchain enthusiasts.

## ü§ù Code of Conduct

By participating in this project, you agree to maintain a respectful and inclusive environment for all contributors.

## üöÄ Getting Started

1. **Fork the Repository**: Click the "Fork" button on GitHub to create your own copy
2. **Clone Your Fork**: `git clone https://github.com/your-username/Empire-of-bits-vorld.git`
3. **Create a Branch**: `git checkout -b feature/your-feature-name`
4. **Install Dependencies**: Follow the setup instructions in the main README
5. **Make Your Changes**: Write clean, well-documented code
6. **Test Your Changes**: Ensure all tests pass and your code works as expected
7. **Commit Your Changes**: Use clear, descriptive commit messages
8. **Push to Your Fork**: `git push origin feature/your-feature-name`
9. **Open a Pull Request**: Submit your PR with a clear description of changes

## üìù Contribution Guidelines

### Types of Contributions

We welcome various types of contributions:

- **Bug Fixes**: Fix issues in existing games or infrastructure
- **New Features**: Add new games, viewer packages, or ecosystem features
- **Documentation**: Improve README, code comments, or API documentation
- **UI/UX Improvements**: Enhance user interface and experience
- **Performance Optimizations**: Improve game performance or blockchain interactions
- **Testing**: Add or improve test coverage

### Game Development

If you're contributing a new game:

1. **Follow the Viewer-Influence Loop**: Ensure your game integrates with Airdrop Arcade for viewer interaction
2. **Package Design**: Create both "Immediate" and "Cycle" influence packages
3. **Documentation**: Document all viewer packages and their effects
4. **Testing**: Test thoroughly with live streaming integration
5. **Cross-Game Compatibility**: Consider how your game fits into the broader ecosystem

### Code Standards

- **Language**: Follow the existing code style and conventions
- **Comments**: Add clear comments for complex logic
- **Naming**: Use descriptive variable and function names
- **Formatting**: Follow the project's formatting rules (use Prettier/ESLint if configured)
- **Type Safety**: Use TypeScript types where applicable

### Commit Messages

Write clear, descriptive commit messages:

```
feat: Add new Thunder Strike package to Space Invaders
fix: Resolve WebSocket connection issue in viewer integration
docs: Update README with new game information
refactor: Optimize game loop performance
test: Add unit tests for viewer package system
```

## üîç Pull Request Process

1. **Update Documentation**: Update README or relevant docs if your changes affect functionality
2. **Add Tests**: Include tests for new features or bug fixes
3. **Ensure Tests Pass**: All existing and new tests should pass
4. **Request Review**: Tag relevant maintainers for review
5. **Address Feedback**: Respond to review comments and make requested changes

### PR Checklist

- [ ] Code follows project style guidelines
- [ ] Self-review completed
- [ ] Comments added for complex code
- [ ] Documentation updated
- [ ] Tests added/updated
- [ ] All tests pass
- [ ] No new warnings introduced
- [ ] Viewer integration tested (if applicable)

## üéÆ Game-Specific Guidelines

### Viewer Package Integration

When adding viewer packages:

- **Immediate Packages**: Should have instant, visible effects
- **Cycle Packages**: Should have duration-based effects with clear start/end
- **Balance**: Ensure packages are fun but not game-breaking
- **Visual Feedback**: Provide clear visual indicators when packages activate
- **Documentation**: Document package effects in game README or docs

### Blockchain Integration

When working with Solana:

- **Error Handling**: Implement robust error handling for blockchain transactions
- **Network Considerations**: Test on both devnet and mainnet (when applicable)
- **Gas Optimization**: Consider transaction costs and optimize where possible
- **Security**: Follow Solana security best practices

## üêõ Reporting Bugs

When reporting bugs, please include:

- **Description**: Clear description of the issue
- **Steps to Reproduce**: Detailed steps to reproduce the bug
- **Expected Behavior**: What should happen
- **Actual Behavior**: What actually happens
- **Environment**: OS, browser, Solana network, etc.
- **Screenshots/Logs**: If applicable

## üí° Suggesting Features

We welcome feature suggestions! When proposing features:

- **Use Cases**: Explain the use case and why it's valuable
- **Implementation Ideas**: Share any ideas for implementation (optional)
- **Integration**: Consider how it fits with existing features
- **Viewer Impact**: If applicable, explain how it enhances viewer interaction
